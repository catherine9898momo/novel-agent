# Novel Agent 主线任务偏航复盘

日期：2026-07-11

## 结论

本次会话最初目标是推进 Novel Agent P0 创作闭环，但实际实现集中在 Career Capture 面试沉淀旁路，Novel Agent 的长短篇选择、素材选择、生成、局部修改、多轮短篇和长篇一致性等核心能力没有前进。

问题不在于 Career Capture 没有价值，而在于它从辅助需求被隐式升级成当前主项目。执行过程中持续优化了局部质量，却没有持续检查这些任务是否推进用户最初要求的产品能力。

## 原始主线

用户希望 Novel Agent 逐步达到以下理想态：

- 支持长篇和短篇选择。
- 用户输入灵感后，从素材库选择和微调故事素材。
- 继续选择文风、人物参考等小说组成元素。
- 生成后支持局部修改，并理解用户修改意图和偏好。
- 短篇能够多轮迭代。
- 长篇在多轮调整和跨章节生成中保持设定与状态一致。

当前 P0 应先交付一条用户可走通的创作垂直切片，而不是先完善旁路基础设施。

## 偏航过程

1. 用户在 P0 方案讨论中增加了“每次优化后自动沉淀面试案例”的辅助需求。
2. Agent 没有先说明该需求会占用一个独立开发周期，也没有请求用户明确决定是否暂停 P0。
3. 用户对触发方式和推荐方案的局部确认，被错误解释成了全局优先级变更。
4. Career Capture 被拆成 pending、index、CLI、Git Hook、并发安全、Skill 和 E2E 六个阶段。
5. 各阶段都通过了测试和独立评审，但没有任务映射到 Novel Agent 的核心创作能力。
6. Career Capture 完成后又触发案例生成、主题索引和 STAR 重构，形成了“记录如何建设记录机制”的元工作递归。

## 根因

### 1. 没有区分主目标、当前里程碑和辅助任务

辅助需求进入对话后，执行计划没有继续显示原始主目标，因此新计划自然取代了旧计划。

### 2. 局部确认被扩大解释

“确认”“继续”“按照推荐执行”只应批准已经明确描述的当前动作，不能自动表示用户同意暂停主线。

### 3. 计划缺少主线贡献映射

任务只描述了技术交付物，没有说明它推进了哪项用户可感知能力。Career Capture 的六个阶段对长短篇、素材选择、局部修改、多轮短篇和长篇一致性的直接贡献均为零。

### 4. 局部质量门替代了方向检查

测试、评审、工作树和并发修复保证了旁路实现质量，但无法证明旁路是当前最值得做的事情。

### 5. 缺少停止条件

连续多个任务没有触碰核心创作链路时，系统没有自动暂停；元工作产生更多元工作时，也没有触发范围检查。

## 造成的成本

- 消耗了大量会话时间和 token。
- 产生了完整但非当前 P0 必需的旁路实现。
- 增加了 Git 分支、测试、文档和待合并状态的管理成本。
- 延迟了用户真正可感知的 Novel Agent 创作能力。

这些成本不能通过“旁路测试很完整”抵消。正确性和优先级是两个独立维度。

## 防偏航规则

### 主目标不可隐式改变

多步骤工作必须持续维护 Primary Goal 和 Current Milestone。无法映射到当前里程碑的任务，默认进入 backlog。

### 优先级调整必须显式确认

切换前必须向用户说明：当前主目标、新任务、会推迟的用户能力、预计成本和 Agent 建议。只有用户明确同意改变优先级，才能暂停主线。

### 局部确认只作用于当前动作

用户说“确认”“继续”“按照推荐执行”，只批准当前已说明的动作，不能推断为全局改优先级。

### 每个任务必须说明主线贡献

实施任务必须写明：用户可感知变化、涉及的核心模块和验收方式。无法回答则不进入当前计划。

### 限制辅助任务预算

未经额外确认，辅助任务或元任务最多占当前里程碑工作量的 15%，或最多一个 commit，先达到者为准。

### 连续偏航自动停止

连续两个任务或 commit 没有推进核心工作流时，必须在第三个任务开始前暂停并报告。

### 禁止元工作递归

不得为了记录、规划或评估辅助机制，继续产生新的记录、规划和评估任务，除非用户明确要求暂停主线。

### Skill 不得扩大范围

Skill 只能服务当前目标。若 Skill 流程会把小任务扩展成新项目，应遵循仓库规则，缩小流程或停止并向用户说明。

## 以后每次阶段汇报的固定内容

- Primary Goal。
- Current Milestone。
- 主线完成度。
- 本阶段用户可感知变化。
- 辅助工作占比。
- 是否发生优先级变化及其确认依据。

## 紧急例外

只有数据丢失、安全漏洞、构建完全阻塞等紧急问题可以临时打断主线。即使属于例外，也必须立即说明原因、影响和恢复主线的条件。

## 恢复方案

1. 冻结 Career Capture 和 STAR 重构，不继续完善元工作。
2. 保留当前旁路成果，但不把它当作 Novel Agent P0 完成度。
3. 回到 P0 技术设计，选择短篇多轮创作闭环作为第一条垂直切片。
4. 优先交付：短篇选择、灵感输入、少量素材卡、文风与人物参考、生成、局部修改、会话级状态保持。
5. 每完成一项核心能力后再使用已有沉淀机制，不再为了沉淀机制创造开发任务。

## 最重要的经验

在开始执行前，先证明任务会推进用户可感知的 Novel Agent 能力，再讨论如何把它实现得完整。测试通过只能证明实现正确，不能证明方向正确。
